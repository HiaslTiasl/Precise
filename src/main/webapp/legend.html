Tasks

A task represents work that is foreseen in certain locations by some craft. Whenever a task is present in a diagram, this means that the corresponding activity has to be executed in all its locations in order to successfully complete the project. Then, we can add more constraints to a task using the concepts of Order and Exclusiveness.

To create a new task, first click and then click at the position in the diagram where you want to create the task. A popup window will appear where you can further specify the task. You must at least specify a task definition by first selecting the phase to which the task belongs and then selecting the task definition. If you are done, click on to confirm and the task will appear at the specified position. You can then select the task by clicking on it, which opens the task's properties on the right. Here you can change the properties, define the task's locations, and specify further constraints on the task (see sections below). When you are done, click to confirm. You can remove a task by selecting it and then clicking on .

Order

By default, the order in which a task is executed in its locations is not specified, any order is fine. To change that, we can define an explicit order to change that. For each attribute of the task's phase, we assign one of four possible order types that define how this attribute contributes to the ordering:
  • NONE: Ignore this attribute (default)
  • PARALLEL: Partition/Parallelize values of this attribute. All following specifications only apply within the same value of this attribute.
  • ASCENDING: Values of this attribute are ordered ascending.
  • DESCENDING: Values of this attribute are ordered descending.
For an example, consider a project with two sectors A and B representing two separate buildings. To express that the task Scaffolding Installation should be performed from bottom to top, we can then set the attribute level to ASCENDING and leave all other attributes to NONE. This means that the first level must be finished in both sectors A and B before we can progress to the next level. However, if A and B are in fact two independent buildings that can be built in parallel, this might be too restrictive. Instead, we want to apply the order on the level only within each building, so for progressing to the next level we do not have to wait until the level is finished in other sectors. We can express this by also setting the attribute sector to PARALLEL

Exclusiveness

Another aspect of tasks is the scope of their exclusiveness. By default, every task is exclusive at unit scope, meaning no to tasks can be performed in one construction unit at the same time. To increase the scope of the exclusiveness of a task, we can explicitly select attributes to which a task is exclusive. If for instance we say that the Task Scaffolding Installation is exclusive at sector and level, once we start that task in a location, no other tasks can be executed in the same level in the same section until Scaffolding Installation finished there.

Alternate Precedence

An alternate precedence is a scoped dependency with an additional constraint: When the source task is finished in a location of the given scope, it must also wait for the target task to be finished there until it can proceed to the next location. The graphical representation indicates this waiting behavior by the cross near the source task. We can think of it as some sort of waiting constraint on the source task, which is indicated by the cross near the source task. In the above example, we have an alternate precedence from Scaffolding Installation to Concrete Pouring at the scope. This means that in each level we must perform Scaffolding Installation before Concrete Pouring, and that Concrete Pouring must be finished in a level in order to start with Scaffolding Installation in another level.

Chain Precedence

A chain precedence is a scoped dependency with an additional constraint: When the source task is finished in the given scope, no other tasks can be performed until the target task is started. By default, dependencies only require that the source must be executed before the target, but they do not prevent other tasks to be executed in between. By using chain precedences, we can change that. In the above example, when we finished Scaffolding Installation in some level, we cannot perform other tasks in that level until we started Concrete Pouring there.
